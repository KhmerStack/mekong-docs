# Deployments

Mekong hosted deployments let you upload a zip archive and serve it from Mekong's hosted infrastructure.

This is different from a live tunnel:

- A live tunnel forwards traffic to your own running machine
- A hosted deployment stores extracted files on the app server and serves them through Mekong's proxy

If your laptop goes offline, a hosted deployment stays online.

## Who can use it

Hosted deployments currently require a paid plan:

- `student`
- `pro`
- `org`

The free plan cannot create hosted deployments.

## Two ways to deploy

### Dashboard

Open the deploy page in the web app:

```text
/dashboard/deploy
```

Workflow:

1. Choose a deploy type
2. Enter the subdomain you want
3. Upload a `.zip`
4. Wait for the deployment to finish
5. Open the live URL from the deployment card

### CLI

You can also deploy from the terminal:

```bash
mekong deploy ./dist
mekong deploy list
mekong deploy stop my-site
```

The CLI packages the directory as a zip, uploads it to the API, and prints the live URL when the hosted deployment is ready.

## What happens on the server

The current hosted deploy flow is:

1. The API receives your zip archive
2. The archive is extracted on the app server into `DEPLOY_DIR/<subdomain>`
3. The app server starts a local static file server for that deployment
4. The app server opens an SSH reverse tunnel to the Mekong proxy server
5. The proxy server claims the deployment subdomain
6. Public traffic is forwarded through the tunnel to the app server file server

This means Mekong no longer relies on `rsync` or per-site Caddy routes for hosted deployments.

## Public URL format

Hosted deployments are served on the configured public wildcard domain:

```text
https://<subdomain>.proxy.mekongtunnel.dev
```

Example:

```text
https://my-site.proxy.mekongtunnel.dev
```

## Logs

The deploy dashboard has a **Logs** button for each hosted deployment.

The current logs view includes:

- deployment summary
- live status
- extracted file list
- recent deployment session logs

If you operate the stack yourself, per-deployment log files are stored on the app server at:

```text
$DEPLOY_DIR/.deploy-logs/<subdomain>.log
```

These logs include:

- tunnel start events
- reconnect/disconnect lines
- SSH session output
- request lines emitted by the running deployment session

## Limits and current behavior

Current hosted deploy limits:

- maximum archive upload: `50 MB`
- active deployments per user: `1`
- one deployment subdomain per active deployment

Current behavior:

- the API waits for the first tunnel handshake before returning success
- active deployments are restored automatically when the API restarts
- stopping a deployment stops the tunnel and removes extracted files
- deleting a deployment also removes the database record

## Supported content today

The current hosted runtime is **static-file oriented**.

What works best:

- plain HTML, CSS, JS
- static site output from generators
- other prebuilt assets that can be served directly from files

Important limitation:

- the hosted deployment flow does **not** start long-running app processes for you
- server-rendered Next.js runtime execution is not part of this hosted path
- PHP execution is not part of this hosted path

So, for now, treat hosted deployments as **static hosting**.

If you need SSR, background processes, or executable PHP, run your own server and expose it with a normal Mekong tunnel instead.

## Troubleshooting

### Deployment says active but the URL fails

Check the deployment logs first from the dashboard.

If you self-host Mekong, also inspect:

- API service logs
- tunnel server logs
- the per-deployment log file on the app server

### Upload returns an internal server error

That usually means one of these failed:

- archive extraction
- local file server startup
- deployment SSH tunnel startup
- deployment subdomain claim

### The dashboard still shows old flow wording

Refresh the frontend after a new deploy. The hosted deployment pipeline now represents:

- archive
- extract
- tunnel
- live

## When to use hosted deployments vs tunnels

Use a hosted deployment when:

- you have a static site or built assets
- you want the site to stay online without your machine running
- you want a simple zip upload flow

Use a normal tunnel when:

- you need a live dev server
- you need SSR or server-side code execution
- you need PHP or another executable runtime
- you want traffic to hit your local machine directly
